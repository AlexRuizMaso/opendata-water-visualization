\chapter{Requisits del sistema}
\label{Ch:Requirements}

\section{Estructura del document}

En aquest capítol es descriuen els requisits que ha de complir el sistema, tant funcionals com no funcionals. Aquesta etapa ha de permetre saber tot allò que cal fer perquè el projecte compleixi tots i cadascun dels seus objectius.

\section{Requisits funcionals}

Els requisits funcionals descriuen les funcionalitats específiques que el sistema ha de proporcionar per a satisfer les necessitats dels usuaris i els objectius del projecte. Aquests requisits es classifiquen per prioritat (Alta, Mitjana, Baixa) i es poden agrupar segons la seva naturalesa.

\begin{itemize}
    \item \textbf{RF-1 (Alta):} Extracció automàtica de dades des de les APIs públiques de la Generalitat de Catalunya. El sistema ha de poder obtenir les dades més recents dels embassaments (nivells, vols i cotas) i de les estacions meteorològiques (precipitació) de forma periòdica sense intervenció manual.
    
    \item \textbf{RF-2 (Alta):} Transformació i neteja de les dades extretes. Les dades obtingudes de les APIs han de ser processades per eliminar inconsistències, convertir units quan sigui necessari, i estructurar-les en un format adequat per a la seva posterior visualització i anàlisi.
    
    \item \textbf{RF-3 (Alta):} Emmagatzematge de dades processades en format JSON versionat. Les dades transformades han de ser desades a fitxers JSON amb timestamps per permetre l'historial de dades i facilitar el control de versions a través de Git.
    
    \item \textbf{RF-4 (Alta):} Desenvolupament d'una interfície web interactiva amb quatre panells de visualització diferenciats:
    \begin{itemize}
        \item Panell 1: Mapa interactiu que mostri la ubicació dels embassaments i estacions meteorològiques amb indicadors en temps real
        \item Panell 2: Evolució temporal dels nivells d'embassaments amb controls de filtratge per periode de temps
        \item Panell 3: Anàlisi de correlació entre dades de precipitació i nivells d'embassaments
        \item Panell 4: Matriu d'alertes de sequera amb indicadors semàfor basats en percentatge d'ocupació
    \end{itemize}
    
    \item \textbf{RF-5 (Mitjana):} Implementació de lògica de detecció automàtica de sequera. El sistema ha de calcular indicadors de risc basats en umbrals predefinits de percentatge d'ocupació dels embassaments (crític <20\%, avís 20-50\%, normal 50-75\%, òptim >75\%) i generar alertes quan es superin els límits establerts.
    
    \item \textbf{RF-6 (Mitjana):} Automatització de l'actualització de dades vía GitHub Actions. El pipeline ETL ha de ser programat per executar-se de forma automàtica a una hora específica cada dia, assegurant que les dades disponibles al frontend siguin sempre les més recents possibles.
    
    \item \textbf{RF-7 (Baixa):} Funcionalitat d'exportació de dades. Permetre als usuaris descarregar les dades visualitzades en formats com CSV o JSON per a un ús posterior en eines externes d'anàlisi.
\end{itemize}

\section{Requisits no funcionals}

Els requisits no funcionals definen les característiques de qualitat, restriccions i condicions que el sistema ha de complir, però que no es relacionen directament amb funcions specifics. Aquests requisits afecten a l'arquitectura general, el rendiment, l'usabilitat i altres aspects del sistema.

\begin{itemize}
    \item \textbf{RNF-1 (Alta): Rendiment.} El sistema ha de ser capaç de processar més de 50.000 registres de dades (correspondents a múltiples dies de mesures d'embassaments i precipitació) en un termini inferior als 5 minuts durant l'execució del pipeline ETL. La interfície web ha de carregar i mostrar les dades en menys de 3 segons per a una experiència d'usuari satisfactory.
    
    \item \textbf{RNF-2 (Alta): Usabilitat.} La interfície ha de ser intuïtiva i accessible per a usuaris amb diversos nivells de competència tècnica, des de professionals del sector fins a ciutadans interessats en el tema. Això inclou navegació clara, etiquetatge comprehensible, i elements interactius que siguin fàcils d'entendre i utilitzar.
    
    \item \textbf{RNF-3 (Alta): Seguretat.} Tot accés a recursos externes (APIs públiques) ha de realitzar-se mitjançant connexions segures (HTTPS). No s'emmagatzemen cap informació sensible com contrasenyes o tokens en el repositori, utilitzant mecanismes com fitxers d'entorn (\texttt{.env}) per a la configuració quan sigui necessari.
    
    \item \textbf{RNF-4 (Mitjana): Portabilitat.} El sistema ha de ser desplegable a diferents plataformes d'hosting sense modificacions significatives al codi, incloent opcions gratuïtes o de baix cost com GitHub Pages, Vercel o Netlify per al frontend, i la capacitat d'executar l'ETL en qualsevol entorn que suporti Node.js.
    
    \item \textbf{RNF-5 (Mitjana): Mantenibilitat.} El codi font ha de seguir conventions establertes, estar ben documentat, i utilitzar una estructura modular que facili la comprensió, modificació i ampliació futura del sistema. Això inclou comentaris adequats, noms de variables i funcions significatius, i separació clara de responsabilitats entre components.
    
    \item \textbf{RNF-6 (Baixa): Escalabilitat.} Mentre que el volúm inicial de dades és manejable, l'arquitectura ha de permetre una augmentació futura del volúm de dades o la addició de nous tipus de dades sense requerir una redesign complet del sistema.
\end{itemize}
